# RMT Output Protocol

`RmtOutputAdapter` реализует протокол `ESP32 Link → STM32 iHeater` для передачи команды нагрева через один цифровой GPIO.

Протокол — импульсно-счётный: число HIGH-импульсов в фрейме кодирует целевую температуру либо команду остановки нагрева.  
Фреймы отправляются непрерывно каждые `framePeriodMs`; STM32 переходит в OFF если не получает валидных фреймов в течение ≥ 3 секунд (watchdog).

Спецификация протокола: [`IHEATER_LINK_PULSE_PROTOCOL.md`](../../../IHEATER_LINK_PULSE_PROTOCOL.md)

## Программное отключение адаптера

Если `outputPin == 0xFF` — адаптер полностью отключён: GPIO не инициализируется, FreeRTOS-таск не создаётся, `isEnabled()` возвращает `false`.

## Параметры тактирования

Делитель тактовой: `clk_div = 80` → 80 МГц / 80 = **1 МГц**, 1 тик = 1 мкс.

Значения по умолчанию (`RmtOutputConfig`):

| Параметр        | Значение    | Описание                                                                     |
|-----------------|-------------|------------------------------------------------------------------------------|
| `syncLowUs`     | 120 000 мкс | Длительность SYNC LOW — маркер начала кадра (STM32 детектирует паузу ≥ 100 мс как старт) |
| `pulseHighUs`   | 2 000 мкс   | Длительность HIGH каждого счётного импульса                                  |
| `pulseLowUs`    | 2 000 мкс   | Длительность LOW после каждого счётного импульса                             |
| `framePeriodMs` | 1 000 мс    | Период повторной отправки фрейма                                             |
| `offCode`       | 10          | Число импульсов, которое STM32 интерпретирует как «остановить нагрев»        |
| `minTempC`      | 45 °C       | Нижняя граница диапазона температуры                                         |
| `maxTempC`      | 80 °C       | Верхняя граница диапазона температуры                                        |
| `stepTempC`     | 1 °C        | Шаг квантования температуры                                                  |

## Структура фрейма

```
______120 мс_______   _2ms_ _2ms_   _2ms_ _2ms_        ______
                   |  |‾‾‾|_     |  |‾‾‾|_     ...     |
___________________|__|   | |____|__|   | |____|  ______|
        SYNC LOW          импульс 1     импульс N   IDLE LOW
```

Линия в idle удерживается в LOW.

1. **SYNC LOW** — линия удерживается LOW в течение `syncLowUs` (120 мс).  
   При текущем периоде 1000 мс SYNC сливается с idle-паузой в одну длинную LOW (~680 мс); STM32 детектирует ≥ 100 мс задолго до первого импульса. 120 мс — гарантированный минимум перед импульсами на случай уменьшения периода.  
   Поскольку поле длительности RMT ограничено 15 битами (≤ 32 767 мкс), 120 мс разбиваются на несколько RMT item с `level = 0`.

2. **N счётных импульсов** — каждый импульс: HIGH `pulseHighUs` + LOW `pulseLowUs` (по 2 мс).  
   STM32 считает нарастающие фронты после SYNC LOW.

3. **End marker** — RMT item с `val = 0`; драйвер завершает передачу, линия возвращается в IDLE LOW.  
   STM32 считает кадр завершённым когда после импульсов наступает следующая длинная пауза.

Максимальная длительность фрейма при `pulseCode = 80`:  
`120 мс + 80 × 4 мс = 440 мс` — укладывается в период 1 000 мс.

## Кодирование команды в pulseCode

```
ControllerOutputMode::Off
    → pulseCode = offCode (10 импульсов = «остановить нагрев»)

ControllerOutputMode::TargetTemperature
    → target    = clamp(round(targetTempC), minTempC, maxTempC)
       step      = stepTempC > 0 ? stepTempC : 5   // защита от деления на ноль
       pulseCode = minTempC + round((target − minTempC) / step) × step
```

`offCode = 10` лежит ниже `minTempC = 45`, поэтому не может возникнуть из температурного диапазона и однозначно трактуется STM32 как остановка нагрева.


## Правила декодирования на STM32

- Пауза ≥ 100 мс → начало нового кадра.
- После паузы STM32 считает нарастающие фронты.
- Следующая длинная пауза → кадр завершён, код интерпретируется как команда.
- Нераспознанный код игнорируется; сохраняется последнее валидное значение.
- Если валидные кадры не приходят ≥ 3 секунд → переход в OFF.

## Отправка

Выделенный FreeRTOS-таск (`rmt_tx`, стек 3072 байт, приоритет `tskIDLE_PRIORITY + 2`) отправляет фрейм каждые `framePeriodMs`. Таск изолирован от main loop: WiFi/MQTT/HA могут подвешивать `loopTask` на секунды, что нарушало бы тайминг RMT.

`rmt_write_items` вызывается с `wait_tx_done = false` — отправка неблокирующая.

Фрейм перестраивается только при изменении `pulseCode` (`builtForCode_ != lastPulseCode_`).

`forceFrame()` будит таск через `xTaskNotifyGive` для немедленной отправки вне расписания.

## Отладочный вывод

```
[RMT] #<seq> t=<ms> dt=<ms> code=<pulseCode> items=<count>
```
